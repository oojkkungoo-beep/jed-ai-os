# ⚖️ Council: GLM + Gemini เป็นเครื่องมือกลางของทีม — 2026-07-16

**คำถาม:** ควรนำ GLM (Z.ai) และ Gemini มาเป็นเครื่องมือให้ agent ในทีมใช้ทำงาน เพื่อคุณภาพเท่าเดิม/ดีขึ้นแต่ประหยัด token Claude หรือไม่ อย่างไร

---

## 🔎 Vera Review — ตรวจ fact ที่ verify ได้

- ✅ **ราคา/แพลน GLM:** มี source ตรวจได้ (z.ai/subscribe, docs.z.ai) — Lite ~$12.60/เดือน (ลด 30%), ตัวฟรีคือ GLM-4.7 Flash + โควตา 20M token ที่ bigmodel.cn
- ⚠️ **ตัวเลข benchmark GLM-5.2 ≠ ตัวที่จะใช้ฟรี:** ผลเทียบ "ใกล้ Opus 4.8" เป็นของ GLM-5.2 (ตัวเสียเงิน) — ยังไม่มีข้อมูล benchmark ที่น่าเชื่อถือของ GLM-4.7 Flash ตัวฟรี → claim "ประหยัดแต่คุณภาพคงเดิม" อยู่ในสถานะ **unverified** ต้องวัดเองจาก pilot
- ⚠️ **คุณภาพ source:** ตัวเลขส่วนใหญ่มาจาก blog เทียบรุ่น/เว็บ pricing บุคคลที่สาม ไม่ใช่ peer-reviewed — Semgrep cyber benchmark เป็นของบริษัทเดียว อย่าใช้อ้างกว้าง
- ✅ **Precedent Gemini ใช้งานได้จริง:** ทดสอบรัน `scripts/gemini_review.py` สำเร็จวันนี้ (ผลอยู่ในส่วน Devil ด้านล่าง)
- 🔧 **พบจุดซ่อม 1 จุด:** `gemini_review.py` ไม่โหลด `.env` เอง ต้อง export ก่อนรัน — ควรให้ Forge เพิ่ม dotenv loading ~3 บรรทัด กันคนรันครั้งถัดไปเจอ error เดิม
- 📌 **แก้ fact ของ Gemini:** Gemini review อ้างถึง HIPAA — ไม่ตรงบริบท Jed อยู่ไทย กฎหมายที่เกี่ยวคือ **PDPA (พ.ร.บ.คุ้มครองข้อมูลส่วนบุคคล)** + ระเบียบหน่วยงานสาธารณสุข แต่สาระความเสี่ยง (ข้อมูลผู้ป่วยห้ามออกนอกระบบที่ควบคุมได้) ยังถูกต้อง

**คะแนน proposal ฉบับแรก: B** — โครงดี มี governance loop แต่ claim ประหยัดยังไม่มีตัววัด และยังไม่มีข้อห้ามข้อมูลอ่อนไหวเขียนเป็นลายลักษณ์อักษร

## 😈 Devil Review — Mode A Bear Case (ใช้ Gemini gemini-2.5-flash เป็นสมองนอกค่าย)

### ท้าทาย (สังเคราะห์จาก Gemini + Devil)
1. **Paradox การตรวจงาน:** GLM-4.7 Flash อ่อนกว่า 5.2 ชัดเจน — ถ้าคุณภาพร่างต่ำพอ token ที่ Claude ใช้ "ตรวจ+แก้" อาจแพงกว่า "ทำเองแต่แรก" → แผนประหยัดกลายเป็นแผนเพิ่มต้นทุน
2. **ระเบิดเวลาข้อมูลอ่อนไหว:** งานของ Jed หลายชิ้นแตะข้อมูลผู้ป่วย/บุคลากร (STAT_OR, Medic_Station, SAR, KM ของโรงพยาบาล) — ส่งเข้า GLM = data วิ่งผ่านเซิร์ฟเวอร์จีน ควบคุมไม่ได้ เสี่ยง PDPA/วินัยราชการ นี่คือจุดที่พังแล้วย้อนกลับไม่ได้ (เคยมีบทเรียน STAT_OR Google Sheet PHI มาแล้ว)
3. **คอขวดรีวิว:** ถ้าทุกงาน GLM ต้องให้ Claude ตรวจละเอียด agent จะกลายเป็น "ช่างซ่อมงานคนอื่น" แทนที่จะสร้างงาน — ประสิทธิภาพทีมรวมอาจลดลง

### Kill-conditions ที่เสนอ (Mode B เสริม)
- ถ้าใน pilot งานจาก GLM ต้องแก้เกิน **50%** ของชิ้นงาน หรือ token ตรวจ+แก้ ≥ token ทำเอง → เลิกใช้กับงานประเภทนั้น
- ถ้าเผลอส่งข้อมูลอ่อนไหวเข้า GLM แม้ครั้งเดียว → หยุดทั้งระบบ ทบทวน process ก่อนใช้ต่อ
- ถ้า Z.ai เปลี่ยนเงื่อนไข free tier/license ให้เก็บ data ไป train โดย opt-out ไม่ได้ → เลิก

**สรุป Devil: ควรแก้ก่อน** — เดินต่อได้แต่ต้องมีรั้วข้อมูล + ตัววัดจริงก่อน ไม่ใช่เชื่อ benchmark คนอื่น

## 🎯 Atlas — มุมมอง CEO

- **Two-way vs one-way door (Bezos):** ตัว integration เป็นประตูสองทาง — script ฟรี ถอดออกได้ทุกเมื่อ ลงทุนต่ำ → ทดลองได้ไม่ต้องกลัว แต่**ข้อมูลอ่อนไหวที่หลุดออกไปเป็นประตูทางเดียว** — ย้อนกลับไม่ได้ ดังนั้นเส้นแบ่งต้องขาดตั้งแต่วันแรก ไม่ใช่ "ค่อยระวังเอา"
- **ทรัพยากรที่แพงจริงของ Jed ไม่ใช่ token — คือเวลา (~1.5 ชม./วัน):** ถ้า GLM ประหยัดเงินแต่เพิ่มรอบตรวจ/รอบสั่งงานให้ Jed หรือทีม นั่นคือขาดทุน กำไรจริงคือ "งานปริมาณมากที่เมื่อก่อนไม่ได้ทำเลยเพราะ token จำกัด" (เช่น retrofit 98 ไฟล์ KM) ถูกปลดล็อก
- **มุมกลยุทธ์:** multi-model เป็นทิศทางอุตสาหกรรม การมีชั้นเครื่องมือ (Claude = สมอง, GLM/Gemini = แรงงาน) ลด vendor lock-in และทีมได้เรียนรู้ก่อนคู่แข่ง — สอดคล้อง Life Vision "ผู้บริหารชีวิต" คือบริหารทรัพยากรเป็นชั้นๆ ไม่ใช่ใช้ของแพงสุดกับทุกงาน
- **คำถามที่ Atlas ขอฝาก Jed:** งานไหนบ้างที่ทุกวันนี้ "ไม่ได้ทำ" เพราะเสียดาย token? นั่นคือตลาดที่แท้จริงของ GLM — ไม่ใช่การย้ายงานที่ Claude ทำดีอยู่แล้ว

## ⚖️ Council — 3 มุม + คำตัดสิน

🔴 **Challenger:** ความเสี่ยงตายตัวเดียวคือข้อมูลอ่อนไหวรั่ว (PDPA/ราชการ) — ที่เหลือ (คุณภาพต่ำ, เสียเวลา) เป็นความเสี่ยงที่วัดได้และถอยได้ อย่าให้ pilot เริ่มโดยไม่มี data boundary เป็นลายลักษณ์อักษร

🔵 **Strategist:** 5-10 ปีข้างหน้า ระบบ AI ส่วนตัวที่อยู่รอดคือระบบที่จัดชั้นโมเดลตามงานเป็น opportunity cost ของการ "ไม่ลอง" คือทีมไม่มีทักษะ multi-model ตอนที่ราคา Claude ยังเป็นข้อจำกัด

🟢 **Executor:** ทำได้จริง — pattern มีแล้ว (gemini_review.py), ของฟรีมีจริง, งาน pilot ชัด (UI dashboard v2 / เอกสารสาธารณะ) วัดผลได้ใน 2 สัปดาห์

**คำแนะนำ: ทำ — แบบ "pilot มีรั้ว" (ไม่ใช่เปิดใช้เต็มระบบทันที)**

**เหตุผล:** เป็น two-way door ที่ลงทุนต่ำและถอยง่าย ความเสี่ยงร้ายแรงจุดเดียว (ข้อมูลอ่อนไหว) ป้องกันได้ด้วยกติกา hard-ban ที่บังคับใช้เชิงโครงสร้าง ส่วน claim เรื่องประหยัดยังพิสูจน์ไม่ได้ → ให้ pilot เป็นคนตอบด้วยตัวเลขจริง ไม่ใช่ blog

**Action:**
1. **Data Boundary (ก่อนติดตั้ง):** เขียนกฎลง CLAUDE.md/team file — ห้ามส่งข้อมูลผู้ป่วย, ข้อมูลบุคลากรระบุตัวได้, ข้อมูลการเงินส่วนตัว, เอกสารราชการภายใน เข้า GLM เด็ดขาด (Gemini ใช้เกณฑ์เดียวกัน) — GLM รับได้เฉพาะ: โค้ด/UI ที่ไม่มีข้อมูลจริง, เอกสารสาธารณะ, เนื้อหาที่ Jed อนุมัติรายชิ้น
2. **ติดตั้ง:** Forge สร้าง `scripts/glm_task.py` (GLM-4.7 Flash ฟรี) + แก้ `gemini_review.py` ให้โหลด `.env` เอง
3. **Pilot 2 สัปดาห์ 3 งาน:** (ก) Forge — ร่าง UI dashboard v2, (ข) Lena — แปลงเอกสาร KM ที่เป็นเอกสารสาธารณะ 5-10 ไฟล์, (ค) Scout — ย่อยเอกสารยาวสาธารณะ 1 ชิ้น — ทุกงานบันทึก: token Claude ที่ใช้ตรวจ+แก้ vs ประมาณการทำเอง, % ที่ต้องแก้
4. **ตัดสินด้วย kill-conditions ของ Devil:** แก้เกิน 50% หรือตรวจแพงกว่าทำเอง → เลิกใช้กับงานประเภทนั้น
5. **Vera ปิด pilot:** สรุปตัวเลขเข้า weekly audit รอบถัดไป แล้ว Council นัดตัดสินรอบสอง (ใช้ต่อ/ขยาย/เลิก)

## 🏁 คำตัดสินสุดท้ายของ Jed (2026-07-16): **รอ — ยังไม่ใช้ GLM ตอนนี้**

หลังฟัง bear case ของ Devil Jed ทบทวนแล้วเห็นว่ายังไม่จำเป็น — เหตุผลหลัก: ไม่มีปัญหาจริงที่ GLM แก้ ณ วันนี้ (solution หาปัญหา), ช่องผู้ช่วยฟรีมี Angie + Gemini อยู่แล้ว, งาน bulk ก้อนใหญ่สุดติดรั้วข้อมูลอ่อนไหวส่งเข้า GLM ไม่ได้อยู่ดี, และความซับซ้อนของระบบคือต้นทุนที่แพงกว่า token

**เงื่อนไขกลับมาเปิดเรื่องนี้ใหม่ (re-entry conditions):**
1. Jed ชน rate limit ของ Claude บ่อยจนงานสะดุดจริง หรือ
2. เกิดงาน bulk ที่ไม่แตะข้อมูลอ่อนไหวก้อนใหญ่ชัดเจน ที่ทุกวันนี้ไม่ได้ทำเพราะเสียดาย token

เมื่อเข้าเงื่อนไข → กลับมาอ่านไฟล์นี้ แล้วเริ่มจากแผน pilot 5 ขั้นของ Council ด้านบนได้เลย ไม่ต้องประชุมใหม่

**ของที่ทำไปแล้วจากวงประชุมนี้:** ซ่อม `scripts/gemini_review.py` ให้โหลด `GEMINI_API_KEY` จาก `.env` เอง (ไม่ต้อง export ก่อนรัน) — ทดสอบผ่านแล้ว
