# 😈 Devil Review: ย้าย .git ออกจากโฟลเดอร์ sync คลาวด์ — Mode B (Polarity Audit)

## สมมติฐาน/ข้อสรุปเดิม
Lena ย้าย git data จริงจาก `D:\NB_G_Drive\Second_Brain\.git` ไปไว้ที่ `D:\Second_Brain_gitdata\Second_Brain.git` (นอกโฟลเดอร์ sync) เพื่อป้องกัน Google Drive sync ชนกับ git internals — สรุปว่า "ปิดความเสี่ยงนี้เรียบร้อยแล้ว"

## ท้าทาย

1. **ตอนนี้ git history ไม่ได้ backup ขึ้นคลาวด์อีกแล้วครับ** — จุดที่ Jed ขอแก้คือ "อยาก backup ขึ้น Google Drive" แต่การย้าย `.git` ออกจาก `D:\NB_G_Drive\` แปลว่า **ประวัติการแก้ไขทั้งหมด (สิ่งที่ git เก็บ) อยู่ใน drive D: เครื่องเดียวเท่านั้น ไม่ sync ไปไหน** ถ้า drive D: พัง จะเหลือแค่ "สถานะล่าสุด" ของไฟล์ (เพราะตัวไฟล์ .md ยัง sync อยู่ในวอลท์) แต่ **ประวัติ/เวอร์ชันเก่าหายหมด** — นี่คือการแลก "ป้องกัน conflict" กับ "เสีย backup coverage ของ history" ซึ่งไม่มีใครพูดเรื่องนี้ตรงๆกับ Jed
2. **`D:\Second_Brain_gitdata\` เป็น single point of failure ใหม่** — ถ้า Jed ลบโฟลเดอร์นี้โดยไม่รู้ว่ามันคือที่เก็บ git จริง (เพราะดูเหมือนโฟลเดอร์เปล่าๆไม่เกี่ยวกับ Second Brain) vault จะกลายเป็น "ไม่มี git" ทันทีโดยไม่มีสัญญาณเตือน เพราะไฟล์ `.git` ใน vault เป็นแค่ pointer บรรทัดเดียว ไม่มีใครสังเกตง่ายๆว่ามันชี้ไปที่อื่น
3. **ยังไม่ทดสอบ failure mode จริง** — สมมติฐานคือ "ย้ายแล้วปลอดภัย" แต่ไม่มีใคร simulate จริงว่าถ้า Google Drive sync ระหว่างที่ vault ไฟล์ .md กำลังถูกแก้ (ไม่ใช่ .git) จะมีปัญหาไหม — ความเสี่ยงเดิมอาจไม่ได้อยู่ที่ `.git` เท่านั้น แต่อยู่ที่ตัวโน้ต .md เองด้วยถ้า obsidian-git auto-commit ทุก 10 นาทีพร้อมกับ Drive sync พอดี

## ถ้าข้อ 1 เป็นจริง
Jed อาจเข้าใจผิดว่า "ตอนนี้มี backup ครบ 2 ชั้นแล้ว (git + cloud)" แต่จริงๆ git history (ของจริงที่มีค่าตอนกู้คืน) อยู่ใน local เท่านั้น ส่วนคลาวด์ backup แค่ "ไฟล์เวอร์ชันล่าสุด" ไม่มีประวัติ — ถ้าไฟล์ .md ถูกแก้ผิดแล้ว sync ขึ้นคลาวด์ไปแล้ว (Drive sync เร็วกว่าที่จะ revert) จะกู้คืนได้จาก git local เท่านั้น ซึ่งก็อยู่ใน drive เดียวกับวอลท์ (D:) — ถ้า D: พังทั้งก้อน ทั้ง git history และวอลท์ปัจจุบันหายพร้อมกัน

## สรุป
**ควรแก้ก่อน** — ไม่ใช่เสี่ยงเกินไปจนใช้ไม่ได้ แต่ Jed ควรรู้ trade-off นี้ตรงๆ และตัดสินใจเอง 2 ทางเลือก:
- (A) ยอมรับว่า git history ไม่ได้ backup คลาวด์ — เน้นว่า Drive sync ป้องกันแค่ "ไฟล์เนื้อหาล่าสุดหาย" ไม่ป้องกัน "ประวัติเก่าหาย" ถ้า D: พังทั้งลูก
- (B) ตั้ง `D:\Second_Brain_gitdata\` ให้ sync ไป Drive ด้วยอีกโฟลเดอร์หนึ่งที่ Drive ไม่ sync แบบ real-time (เช่น sync ทุกคืนตอนไม่มีการเขียนไฟล์ หรือ backup เป็น zip รายสัปดาห์) — ได้ทั้งกันชนและกันพังไปพร้อมกัน
